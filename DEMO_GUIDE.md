# 🚀 CoAct Demo & Participation Guide

This document contains essential precautions, warnings, and best practices for the Project Head to ensure a smooth live demonstration of the CoAct platform.

---

## 🎙️ Voice & Audio Precautions
The real-time voice system is the core of CoAct. Please follow these rules strictly:

### 1. HTTPS is Mandatory
WebRTC (the technology behind the voice chat) **requires a secure context**.
- **Localhost**: Works fine on `http://localhost:3000`.
- **Remote Access**: If people are joining via a URL, it **MUST** be `https://`. Voice will fail on `http://` for security reasons.

### 2. Browser Permissions
- Participants must **Allow** microphone access when prompted.
- **Chrome/Edge/Safari** are recommended. Some in-app browsers (like within WhatsApp or Instagram) may block microphone access. Encourage users to open the link in a dedicated browser.

### 3. Feedback Loops (Larsen Effect)
If multiple people are in the same physical room participating:
- **USE HEADPHONES**: This is critical. Without headphones, microphones will pick up the speaker output of others, causing a loud, high-pitched screeching feedback loop.
- **Mute when not talking**: Use the toggle button frequently if not using headphones.

## 🎮 Game-Specific Warnings

### Word Chain
- **Dictionary Check**: The game uses an external API (`dictionaryapi.dev`) to validate words. If the internet is slow or the API is down, it might take a second to validate or fallback to allowing all words.
- **Timer**: Every player has **15 seconds**. If they don't answer, the turn automatically skips. Warn players to be quick!

### Antakshari / Voice Games
- **Speech Recognition**: The Antakshari mode uses the Browser's Speech Recognition API (Webkit). 
  - **Browser**: This works best in **Google Chrome**. It may not work in Firefox or some mobile browsers.
  - **Language**: The recognition is currently optimized for **Hindi (`hi-IN`)**. Sing clearly and avoid background noise for better transcriptions.
- **Auto-Moderation**: The system automatically mutes/unmutes based on turns. **Do not manually try to unmute** if it's not your turn, as the host logic might override it.

## 🛡️ Moderation for the Host
As the Demo Lead (Host), you have ultimate control:

- **Mute All**: Use this in the "Moderation" panel if the room gets too noisy.
- **Kick User**: If a random participant joins and causes trouble, you can remove them instantly from the "Moderation" panel.
- **Unmute All**: Use this to allow everyone to cheer or collaborate during study/focus modes.

---

## 📱 Device & Connection Stability

### 1. Battery & Heat
Real-time voice and high-quality animations can be demanding on older mobile devices. Advise participants to ensure their phones are charged.

### 2. Connection Drops
If a participant's internet drops:
- They can simply **Refresh the page**.
- The server will recognize their name and reconnect them to their active session and team automatically.

### 3. Maximum Participants
While the system is robust, for the best experience during a demo, try to keep active game participants (like Ludo or Uno) within the recommended limits (2-4 for Ludo, up to 10 for Uno). Use **Spectator Mode** for larger crowds.

---

## 💡 Pro-Tip for the Demo
Before the audience joins:
1. Open the **Moderation Panel** (Shield icon).
2. Familiarize yourself with the **"Mute All"** and **"Unmute All"** buttons.
3. If you are demoing in a hall, ensure YOUR device is connected to the sound system, and all other participants are on **low volume or headphones**.
